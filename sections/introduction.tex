The rapid growth of deep learning workloads has driven a proliferation of specialized tensor accelerators in both academia and industry. Designs such as Gemmini~\cite{gemmini-dac2021}, MAERI~\cite{maeri-asplos2018}, SIGMA~\cite{sigma-hpca2020}, Eyeriss~\cite{eyeriss-jssc2017}, NVDLA~\cite{nvdla}, and FEATHER~\cite{tong2024feather} explore diverse microarchitectural innovations—systolic arrays, reconfigurable interconnects, flexible dataflow engines—to deliver orders-of-magnitude improvements in performance and energy efficiency over general-purpose processors. Meanwhile, a small number of platforms backed by major vendors, notably Google's TPU~\cite{tpuv1} with XLA~\cite{xla}, Intel AMX~\cite{amx} with oneDNN~\cite{oneDNN_Contributors_oneAPI_Deep_Neural}, and NVIDIA GPUs with CUDA/cuDNN~\cite{nickolls2008scalable, chetlur2014cudnn}, enjoy mature compiler toolchains that connect them seamlessly to frameworks like PyTorch~\cite{paszke2019pytorch}, JAX~\cite{jax2018github}, and TensorFlow~\cite{tensorflow2015-whitepaper}. The contrast is stark: as Table~\ref{tab:accel-landscape} summarizes, accelerators with mature, end-to-end compiler support number fewer than five, while over a dozen open-source designs lack any usable compiler backend. These accelerators can only be exercised through hand-written kernel libraries that cover a narrow set of operators and miss cross-layer fusion opportunities, leaving the vast majority of proposed hardware unable to run real ML models end-to-end.

The root cause of this software deficit lies upstream of the compiler: the ISA semantics of most accelerators have never been formally defined. In the CPU world, the ISA serves as the foundational contract between hardware and software—compilers, simulators, and verification tools are all built on top of it. For accelerators, however, ISA information is typically scattered across RTL source code, informal documentation, or exists only in the minds of hardware designers. Huang et al.\ formalized this observation with the Instruction-Level Abstraction (ILA)~\cite{ila}, noting that the accelerator community lacks both a common framework for specifying accelerator behavior and a unified way to reason about programs that interact with accelerators. From the programming language side, Exo~\cite{exo} demonstrated that proprietary hardware interfaces force each company to maintain its own compiler fork—a cost that discourages hardware innovation and entrenches legacy platforms whose dominance stems from ecosystem lock-in rather than architectural merit. Figure~\ref{fig:overview} summarizes this landscape: the top-down ACT ecosystem assumes a hand-written TAIDL specification, while the bottom-up ILA framework produces only scalar-level models, leaving no automated path from RTL to tensor-level semantics.

Recent work has made significant strides in closing the gap from ISA specification downward. TAIDL~\cite{taidl-micro2025}, presented at MICRO 2025, introduced the first domain-specific language for defining tensor accelerator ISAs and showed how test oracles can be auto-generated from these definitions, achieving orders-of-magnitude speedups over hand-crafted functional simulators like Gemmini Spike. Building on TAIDL, the ACT project~\cite{act-arxiv} demonstrated that sound and complete compiler backends can be automatically generated from ISA descriptions alone, producing code that matches or outperforms hand-optimized kernel libraries. Together, these efforts establish a compelling pipeline from ISA specification to ML framework integration: given a TAIDL definition, ACT generates a compiler backend that plugs into XLA, enabling end-to-end execution from JAX or PyTorch. However, this pipeline has a critical prerequisite: someone must first write the TAIDL specification by hand—a process that requires deep understanding of the accelerator's architectural state, instruction encoding, and tensor-level semantics.

This manual specification bottleneck is not a theoretical concern but a well-documented one. Writing formal accelerator models---whether scalar ILAs or tensor-level TAIDL definitions---has been characterized in prior work as ``tedious and error-prone''~\cite{autogenilaiccad} and as ``a significant burden for accelerators which may not have the volume or life-span compared to processors''~\cite{autogeniladate}; the ILA formalism itself notes that model construction is either synthesized through expertise-intensive templates or, as in the AES case study, ``manually written'' in a dedicated Python library~\cite{ila}. On the hardware side, RTL designs iterate rapidly as architects explore microarchitectural trade-offs; on the software side, TAIDL and ACT can automate everything downstream of the ISA spec but cannot function without one. The result is that building compiler and simulator support still involves considerable manual effort, slowing innovation and widening the gap between hardware and software research communities. The 3LA methodology~\cite{3la} illustrates the same pattern at the application level: it explicitly requires two hand-authored inputs per accelerator---an ILA model and a set of IR-to-accelerator mapping rules---and even with the ILA formal framework available, testing prototype accelerator designs on real applications end-to-end remains prohibitively difficult due to the cost of constructing specialized compiler and simulator support. This bottleneck also limits the potential of emerging LLM-based agents for EDA~\cite{zhong2024llm4eda, wu2024chateda, liu2024chipnemo, patra2024aieda}, which have shown that structured, machine-readable hardware semantics are critical for agent effectiveness~\cite{hong2025autocomp,nie2026kernelcraft,yu2025spec2rtl}, for most accelerators, simply do not exist in any structured form.

In this paper, we present TensorLift, the first end-to-end MLIR-based pipeline that lifts RTL-extracted accelerator semantics to TAIDL-like tensor ISA specifications. Building on prior architecture-level model extraction~\cite{autogeniladate, autogenilaiccad} that produces bit-level MLIR from flattened RTL, TensorLift employs an 8-pass MLIR semantic lifting pipeline that progressively recovers high-level tensor structure from low-level arithmetic operations, as illustrated in Figure~\ref{fig:code}. The passes perform sign-extension chain canonicalization, multiply-accumulate (MAC) pattern detection, control-flow specialization, hardware saturation/clamp recovery, loop reconstruction from unrolled MAC chains, and linalg-level lifting---bridging the abstraction gap between gate-level hardware and the tensor-level semantics required by tools like TAIDL and ACT. A final metadata emission pass annotates the lifted MLIR with structured \texttt{taidl.*} attributes, enabling a generic assembler to produce complete TAIDL specifications automatically.

We evaluate TensorLift on the Gemmini systolic-array accelerator (16$\times$16, INT8), extracting all 11 hardware instructions across three controllers (Execute, Load, Store). The lifting pipeline achieves a 92.9\% reduction on processing elements (686 to 49 lines, of which only 17 encode the semantic computation core), where bit-manipulation canonicalization collapses 383 sign-extension operations, and 30--40\% on controller modules whose remaining lines encode irreducible control logic (FSMs, address computation, muxing). Our bottom-up extraction also reveals that 4 of the 15 entries in TAIDL's hand-written Gemmini specification---softmax, GELU, layernorm, and fused add-with-transpose---are software-composed macro-instruction sequences rather than hardware ISA primitives, a distinction that TensorLift naturally surfaces. Furthermore, TensorLift discovers hardware features omitted from the hand-written reference, including pooling engine semantics (reduce-max with saturation), im2col hardware support (reshape followed by dot product), and multi-bank DMA configuration with bank selection from instruction encoding bits. Correctness is validated through Z3~\cite{z3} SMT equivalence proofs and Gemmini Spike simulator testing, with 10 of 10 generated instruction oracles character-identical to the hand-written reference. We additionally confirm the pipeline's generality on TVM's VTA tensor processor: the same passes lift all four VTA datapath modules (TensorGemm, TensorAlu, Store, and the Load command generator) without accelerator-specific changes, with extracted semantics formally verified equivalent to the underlying RTL.

By filling the frontmost gap in the RTL-to-framework pipeline, TensorLift enables an automated path from RTL to ML framework integration, as illustrated in Figure~\ref{fig:overview}. The complete pipeline---RTL through TensorLift to ISA specification, then through ACT to a compiler backend plugged into XLA---removes the need to hand-write the tensor-level ISA model, the step that has historically been the dominant bottleneck.

In summary, this paper makes the following contributions:

\begin{itemize}
\item We propose the first end-to-end MLIR-based pipeline that lifts RTL-extracted accelerator semantics to TAIDL-like tensor ISA specifications, addressing the manual specification bottleneck that currently blocks the adoption of automated compiler generation tools such as TAIDL and ACT.
\item We design an 8-pass MLIR semantic lifting pipeline that progressively recovers high-level tensor structure from bit-level arithmetic: sign-extension canonicalization, MAC pattern detection, control-flow specialization, saturation/clamp recovery, loop reconstruction, linalg lifting, and structured TAIDL metadata emission.
\item We evaluate TensorLift on the Gemmini systolic-array accelerator, achieving 100\% hardware instruction coverage (11/11 instructions, 3 controllers, 127 extracted MLIR files). The lifting pipeline collapses the extracted bit-level MLIR by up to 92.9\% on processing elements (686 $\to$ 49 lines, of which only 17 encode the tensor computation core) and by 30--40\% on controller modules, with the residue being irreducible control logic (FSMs, address arithmetic, and instruction muxing). TensorLift discovers hardware features absent from the hand-written reference---pooling engine semantics, im2col hardware support, and multi-bank DMA---and distinguishes true hardware ISA from software-composed macro-instructions. Correctness is validated via Z3 SMT proofs and Spike simulator testing. We further confirm generality by running the same pipeline on TVM's VTA tensor processor, where it lifts all four VTA datapath modules without accelerator-specific changes.
\item We close the loop from RTL to ML framework integration: starting from the RTL source files and a top-module name, TensorLift produces a TAIDL specification that plugs directly into the ACT ecosystem for automatic compiler backend generation, removing the manual ISA-specification step that has historically been the dominant bottleneck.
\end{itemize}
