The rapid growth of deep learning workloads has driven a proliferation of specialized tensor accelerators in both academia and industry. Designs such as Gemmini~\cite{gemmini-dac2021}, MAERI~\cite{maeri-asplos2018}, SIGMA~\cite{sigma-hpca2020}, Eyeriss~\cite{eyeriss-jssc2017}, NVDLA~\cite{nvdla}, and FEATHER~\cite{tong2024feather} explore diverse microarchitectural innovations—systolic arrays, reconfigurable interconnects, flexible dataflow engines—to deliver orders-of-magnitude improvements in performance and energy efficiency over general-purpose processors. Meanwhile, a small number of platforms backed by major vendors, notably Google's TPU~\cite{tpuv1} with XLA~\cite{xla}, Intel AMX~\cite{amx} with oneDNN~\cite{oneDNN_Contributors_oneAPI_Deep_Neural}, and NVIDIA GPUs with CUDA/cuDNN~\cite{nickolls2008scalable, chetlur2014cudnn}, enjoy mature compiler toolchains that connect them seamlessly to frameworks like PyTorch~\cite{paszke2019pytorch}, JAX~\cite{jax2018github}, and TensorFlow~\cite{tensorflow2015-whitepaper}. The contrast is stark: as Table~\ref{tab:accel-landscape} summarizes, accelerators with mature, end-to-end compiler support number fewer than five, while over a dozen open-source designs lack any usable compiler backend. These accelerators can only be exercised through hand-written kernel libraries that cover a narrow set of operators and miss cross-layer fusion opportunities, leaving the vast majority of proposed hardware unable to run real ML models end-to-end.

The root cause of this software deficit lies upstream of the compiler: the ISA semantics of most accelerators have never been formally defined. In the CPU world, the ISA serves as the foundational contract between hardware and software; for accelerators, however, ISA information is typically scattered across RTL source code, informal documentation, or exists only in the minds of hardware designers~\cite{ila, exo}. Figure~\ref{fig:overview} summarizes this landscape: the top-down ACT ecosystem assumes a hand-written TAIDL specification, while the bottom-up ILA framework produces only scalar-level models, leaving no automated path from RTL to tensor-level semantics.

Recent work has made significant strides in closing this gap from ISA specification downward. TAIDL~\cite{taidl-micro2025} introduced a domain-specific language for defining tensor accelerator ISAs, and ACT~\cite{act-arxiv} showed that compiler backends can be automatically generated from such specifications (\S\ref{subsec:spec-bottleneck}). However, this pipeline has a critical prerequisite: someone must first write the TAIDL specification by hand---a process that requires deep understanding of the accelerator's architectural state, instruction encoding, and tensor-level semantics.

This manual specification bottleneck is well-documented: writing formal accelerator models has been characterized as ``tedious and error-prone''~\cite{autogenilaiccad} and as ``a significant burden for accelerators which may not have the volume or life-span compared to processors''~\cite{autogeniladate}. On the hardware side, RTL designs iterate rapidly; on the software side, TAIDL and ACT automate everything downstream of the ISA spec but cannot function without one. Prior efforts such as ILA~\cite{ila} and 3LA~\cite{3la} formalized accelerator interfaces but still require hand-authored models, and LLM-based EDA agents~\cite{zhong2024llm4eda, wu2024chateda, liu2024chipnemo, patra2024aieda} that depend on machine-readable hardware semantics~\cite{hong2025autocomp,nie2026kernelcraft,yu2025spec2rtl} cannot operate when such semantics do not exist.

In this paper, we present ATLAAS, the first end-to-end MLIR-based pipeline that lifts RTL-extracted accelerator semantics to TAIDL-like tensor ISA specifications. Building on prior architecture-level model extraction~\cite{autogeniladate, autogenilaiccad} that produces bit-level MLIR from flattened RTL, ATLAAS employs an 8-pass MLIR semantic lifting pipeline that progressively recovers high-level tensor structure---from bit-manipulation canonicalization through idiom detection and loop reconstruction to structured metadata emission---bridging the abstraction gap between gate-level hardware and the tensor-level semantics required by tools like TAIDL and ACT (Figure~\ref{fig:code}).

We evaluate ATLAAS on two open-source accelerators: Berkeley's Gemmini (systolic array, 16$\times$16, INT8) and TVM's VTA. The lifting pipeline achieves up to 92.9\% reduction on compute-dominated modules, discovers hardware features omitted from hand-written references, and generalizes across both accelerators without modification. All lifted semantics are formally verified equivalent to the RTL via Z3~\cite{z3} SMT proofs. Fed into ACT, the extracted specification yields a compiler backend that achieves a geometric mean speedup of $1.014\times$ over hand-written Gemmini kernels on workloads including ResNet-50 and MobileNet.

By filling the frontmost gap in the RTL-to-framework pipeline, ATLAAS enables an automated path from RTL to ML framework integration, as illustrated in Figure~\ref{fig:overview}. The complete pipeline---RTL through ATLAAS to ISA specification, then through ACT to a compiler backend plugged into XLA---removes the need to hand-write the tensor-level ISA model, the step that has historically been the dominant bottleneck.

In summary, this paper makes the following contributions:

\begin{itemize}
\item We propose the first end-to-end MLIR-based pipeline that lifts RTL-extracted accelerator semantics to TAIDL-like tensor ISA specifications, addressing the manual specification bottleneck that currently blocks the adoption of automated compiler generation tools such as TAIDL and ACT.
\item We design an 8-pass MLIR semantic lifting pipeline that progressively recovers high-level tensor structure from bit-level arithmetic through four phases: canonicalization, idiom detection, loop reconstruction, and metadata emission.
\item We evaluate ATLAAS on Gemmini and VTA, achieving up to 92.9\% code reduction on compute modules, discovering hardware features absent from hand-written references, and formally verifying all lifted semantics via Z3 SMT proofs. The same pipeline generalizes across both accelerators without modification.
\item We close the loop from RTL to ML framework: we contribute HLO frontend support, convolution pattern matching, and multi-layer memory allocation to ACT, enabling end-to-end compilation from the extracted specification to executable binaries that achieve $1.014\times$ geometric mean speedup over hand-written Gemmini kernels on workloads including ResNet-50 and MobileNet.
\end{itemize}
